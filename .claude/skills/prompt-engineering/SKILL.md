# prompt-engineering — プロンプト作成リファレンス

プロンプトを書く・書き直す・レビューする場面で「プロセスではなく到達点を書く」姿勢と「簡潔・明快・一意」の3原則を当てはめ、伝わる最短のプロンプトに整えるためのリファレンス。

## 基本姿勢

伝えるべきことは伝えつつ、できる限り短く。

## 適用範囲 — 書き手向けであり、プロンプト本文には書き込まない

本リファレンスの原則・観点は **プロンプトを書く人** が従うものであり、生成するプロンプト本文に書き込む指示ではない。生成・編集したプロンプトに「簡潔に」「DRY で」「冗長を避けて」「明快に」などプロンプト設計のメタ的視点を混入させない。読み手にとってノイズとなり、依頼内容そのものを薄める。

- 悪い例: 「次のコードを簡潔かつ DRY にリファクタリングして。冗長な記述は避けて。」
- 良い例: 「次のコードをリファクタリングして。重複ロジックは関数に切り出す。」

## プロセスではなく到達点を書く

依頼の本質は「何を達成したいか」。手順をなぞらせるよりも、**ゴール・成功条件・制約・利用可能なエビデンス** を定義し、解決経路はモデルに委ねた方がよい。過剰な手順指示はノイズとなり、本当に守るべき制約を埋もれさせる。

- 悪い例: 「まず `A.py` を開いて、`X` 関数を探して、引数を確認して、呼び出し元を grep して、一つずつ修正して...」
- 良い例: 「`A.py` の `X(...)` の呼び出し元を `flag=True` で統一する。既存の挙動を変えないこと。」

### 強調語は絶対のルールにだけ使う

過剰な手順指示と同じく、強調語の乱発も本質を埋もれさせる。「できれば守ってほしい」レベルは通常の指示として書き、**破ったら成果物として認められない** ものだけ ALWAYS / NEVER / MUST を使う。

- 悪い例: 「ALWAYS キャメルケース、NEVER 80文字超え、MUST 早期 return、ALWAYS テストを書く...」
- 良い例: 「関数名はキャメルケース、1行80文字以内、早期 return を優先。NEVER 既存テストを削除しない。」

## 3つの原則

### 1. 簡潔 — 必要なことを必要なだけ

冗長な表現・前置き・装飾語を削る。同じ意味の語を繰り返さない。

- 悪い例: 「お手数ですが、もし可能であれば、この関数の動作について簡単に説明していただけると幸いです。」
- 良い例: 「この関数の動作を説明して。」

### 2. 明快 — 何を求めているかが一目でわかる

依頼・条件・期待する出力形式を、読み手が探さずに掴める順序と構造で並べる。長くなるなら箇条書きや見出しで区切る。

- 悪い例: 「ログを見て、エラーの傾向と頻度と、あとできれば対策の案も、JSON で返してね。」
- 良い例:
  > 添付ログから次を JSON で返す。
  > - errors: エラー種別ごとの件数
  > - top_3_actions: 推奨対応策（上位3件）

### 3. 一意 — 解釈の余地を残さない

「適切に」「いい感じに」「よろしく」など読み手判断に委ねる語を排除する。対象・範囲・条件を具体化する。代名詞は曖昧にしない。

- 悪い例: 「このファイルをいい感じにリファクタリングして。」
- 良い例: 「`utils.py` の `parse_log` 関数を副作用のない pure function に書き換えて。シグネチャは変えない。」

## 3原則の関係

到達点（何を伝えるか）が固まった上で、表現（どう伝えるか）として3原則を当てる。3原則同士が衝突したら **一意 > 明快 > 簡潔** の順で優先。誤解されるくらいなら長くなってよい。読みづらいくらいなら短さを諦めてよい。

## 使い方

書こうとしている／書いたプロンプトを次の順で点検する。

1. **到達点** — 手順を書きすぎていないか。ゴール・成功条件・制約に置き換えられないか。強調語が乱発されていないか。
2. **一意** — 曖昧語・代名詞・「適当に」系の語が残っていないか。
3. **明快** — 依頼と出力形式が一目でわかる構造か。
4. **簡潔** — 装飾・繰り返し・無駄な前置きは削れないか。

引っかかった箇所だけ書き直す。原則を満たすために原文を全面的に書き換える必要はない。
