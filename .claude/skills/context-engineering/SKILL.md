# Context Engineering — 設計リファレンス

Claude Code プロジェクトにおけるコンテキスト（CLAUDE.md・サブエージェントプロンプト・スキル・ドキュメント等）の配置判断のための設計原則集。

## 使い方

ユーザーの相談内容に該当する `references/*.md` のみを読み込み、その判断基準に沿って出典（ファイル名）を明記して回答する。全ファイルを一度に読む必要はない。

## 3つの設計原則（優先順位順）

Claude Code のコンテキストエンジニアリング全体を貫く3つの原則。優先順位は記載順。

### 1. DRY 原則（コンテキストの重複最小化）

同じ情報を複数の場所に記述しない。ある情報は1つの場所にのみ定義し、他からは参照によってアクセスする。

#### 適用例

- 複数のエージェントが共有する手法は、各エージェントプロンプトに重複して書かず、スキルとして1箇所に切り出す。
- CLAUDE.md とドキュメントで同じ情報を持たない。詳細はドキュメントに置き、CLAUDE.md からはパスで参照する。
- サブエージェントごとの責務に含まれる共通手法は、各サブエージェントプロンプトに書かず、共有スキルに集約する。

### 2. 最小コンテキスト原則

必要なコンテキストだけを、必要なときにロードする。全情報を常にロードするのではなく、タスクや役割に必要なコンテキストのみを取り込む。

この原則は静的なロード戦略だけでなく、動的な作業の配置にも及ぶ。

#### 静的側面（ロード対象の絞り込み）

- CLAUDE.md は常に薄く保つ（全エージェントが常に参照するため）。
- サブエージェント固有の情報は CLAUDE.md に入れず、サブエージェントプロンプトに閉じる。
- スキルは名指しで使用されるタイミングでのみロードされる。

#### 動的側面（作業の委譲によるコンテキスト節約）

長時間タスクで原則を維持するための適用結果として、スキルを使った具体的な作業はサブエージェントに委譲し、メインエージェントはオーケストレーションに専念する。サブエージェントの独立コンテキストがこの委譲を支える仕組み（詳細は `references/agents.md` を参照）。

### 3. 疎結合原則

呼び出されるスキルやエージェント、およびエージェント同士は疎結合であるべき。各要素は独立して定義・変更可能であり、特定の組み合わせに依存しない。

#### 適用例

- スキルはエージェントに従属しない。独立した共有リソースとして存在する。
- サブエージェントが使うスキルは個別宣言が必要（詳細は `references/skills.md` を参照）。
- ワークフロースキルの追加・変更が気軽にできる。How スキルは複数のワークフローから再利用できる。

### 原則の優先順位

3原則が衝突する場合は、記載順（DRY → 最小コンテキスト → 疎結合）で上位を優先する。

## ファイルパスの記法

CLAUDE.md・サブエージェントプロンプト・スキル・ドキュメント・中間生成物において、他ファイルへの参照パスはプロジェクトルートからの相対パスで表記する。

- `docs/architecture.md` ✅
- `plugins/claude-code-workflow-kit/skills/context-engineering/SKILL.md` ✅
- `./architecture.md`、`../docs/architecture.md` ❌（参照元の階層に依存する）
- `/Users/.../docs/architecture.md` ❌（環境依存の絶対パス）

参照元の階層・作業ディレクトリ・実行環境に依らず一意に解釈できるため。

例外：単一スキル内に閉じる `references/` 間の相互参照は、スキルの可搬性を保つため basename を許容する（本スキル内で `agents.md` と書くなど）。スキル外を指すパスは本ルールに従う。

## トピック別リファレンス

相談内容に応じて該当ファイルを読み込む。

- 全体像・関係図・タスクの流れ — `references/overview.md`
- エージェント（メイン／ユーザー定義サブ／組み込みサブ）の設計 — `references/agents.md`
- CLAUDE.md に何を書くか／書かないか — `references/claude-md.md`
- サブエージェントプロンプトの設計 — `references/subagent-prompts.md`
- スキルの分類（ワークフロースキル／How スキル）と切り出し判断 — `references/skills.md`
- ソースコードとドキュメントの位置づけ — `references/code-and-docs.md`
- 中間生成物・作業中の記憶（揮発系コンテキスト） — `references/ephemeral.md`

## 判断に迷ったとき — 配置先の決定マトリクス

「この情報はどこに置くべきか？」は、対象範囲（誰が必要とするか）と情報種別（What/Why か、How か、地図か、揮発か）の2軸で判定する。

| 対象範囲 | 配置先（種別ごと） |
|---|---|
| 全エージェント共通 | What/Why は CLAUDE.md。地図は CLAUDE.md からパス参照し本体はドキュメント |
| 複数サブエージェント共有 | What/Why と地図はドキュメント。How は How スキル |
| 単一サブエージェント固有 | What/Why と How はサブエージェントプロンプト |
| 人間が起点として呼び出す定型作業 | ワークフロースキル |
| タスク実行中のみ | 中間生成物 / 作業中の記憶 |

軸の判定に迷う典型ケース。

- 2つのサブエージェントだけが必要とする How — How スキルへ切り出すのが基本（DRY 原則）。重複定義の許容範囲は `references/skills.md` の判断基準を参照。
- 全エージェント共通だが詳細が長い — CLAUDE.md には要約とパスのみ、詳細はドキュメントに置く。
- 再利用は今は無いが他プロジェクトに持っていきたい How — 単一サブエージェント利用でもスキルとして切り出す選択肢がある（`references/skills.md` の判断基準を参照）。

判定できない場合は、対応するトピック別リファレンスを読み込み、そこに書かれた判断基準に従う。
