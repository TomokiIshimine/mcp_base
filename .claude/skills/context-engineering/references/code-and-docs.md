# ソースコードとドキュメント

タスクの結果として永続的に残るプロジェクトの資産。

## ソースコード — 真実の源泉

メインエージェント・サブエージェントの両方から参照・更新される。プロジェクトにおける最も信頼すべき情報源であり、「コードが真実のドキュメントである」という思想の中核。

## ドキュメント — コード探索のための地図

メインエージェント・サブエージェントの両方から必要に応じて参照・更新される。

### 位置づけの思想

- **コードが真実のドキュメントである** — ドキュメントはコードの代替ではない。ドキュメントが増えるとコードとの乖離が起きやすく、メンテナンスコストが高いため、補助的な役割にとどめて少なく保つ。
- **ドキュメントは「地図」である** — コード全体を毎回読むコストを下げ、全体像を把握した上で最適なコードを探索するためのガイドとして機能する。
- **更新され続けることが前提** — マイルストーンや機能実装完了時など、定期的なタイミングでコードに追従して更新される。

### 判断基準

以下のいずれかに該当する情報は、ドキュメントとして管理する。

- **全体の地図としての役割を持つ情報** — コード全体を毎回読むことなく、全体像を把握し最適なコードを探索するために必要な情報
- **頻繁に更新される情報** — プロジェクトの進行に伴い変化し続ける情報は、コードに散在させるよりもドキュメントとして一元管理する方がメンテナンスしやすい

### スキルとの区別

スキルはドキュメントを参照しない（`overview.md` の関係図を参照）。スキルは自己完結した再利用可能な How であり、プロジェクト固有のドキュメントに依存させない。

- **スキル** — プロジェクト横断で再利用可能な「どうやるか」
- **ドキュメント** — このプロジェクト固有の「地図」や「設計判断」

## CLAUDE.md との関係

CLAUDE.md 自体はドキュメントの一種ではなく、全エージェントが常に参照する共通基盤という位置づけ。CLAUDE.md からは各ドキュメントへのパスを明示してナビゲーションを提供するが、詳細はドキュメント側に置く（DRY 原則）。

ドキュメント内に記載するパス記法は SKILL.md「ファイルパスの記法」を参照。
