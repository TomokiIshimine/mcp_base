# スキル — 再利用可能な技術的能力

メインエージェント・サブエージェントの両方から必要に応じて使用される再利用可能なコンテキスト。

## スキルの2類型

スキルは用途によって次の2類型に分かれる。

### ワークフロースキル

各種定型作業の起点として、人間がエージェントに対して呼び出すスキル。機能実装、リリース、バグ調査などの定型的なタスクごとに、遂行すべき手順や呼び出すべきエージェント・How スキルの構成を定義する。

- 例 — 機能実装ワークフロー、リリースワークフロー、バグ調査ワークフロー
- 呼び出し元 — 人間（ユーザー）
- 役割 — タスクの起点として、以降の流れ全体を規定する

### How スキル

特定の作業を行う際の具体的な手法・注意点を定義するスキル。エージェントが判断・実装を行う際、あるいはワークフローの各ステップ内で参照される技術的なリファレンスとして機能する。

- 例 — フロントエンドのコーディング手法、バックエンドのコーディング手法、テスト作成の指針
- 呼び出し元 — エージェント、またはワークフロースキル内の手順
- 役割 — 特定の作業領域における「どうやるか」を規定する

## スキル共通の重要な特性

- スキルはエージェントに従属しない。独立した共有リソースとして存在する。
- 複数のサブエージェントから共有して使用される。
- サブエージェントは親エージェントで使用中のスキルを自動継承しない。各サブエージェントが使用するスキルは、そのサブエージェントプロンプト内で個別に宣言する必要がある。これにより「どのサブエージェントがどのスキルを使うか」の依存関係はサブエージェント側の宣言で閉じ、疎結合原則と整合する。

## 起動方法の前提

エージェントによる自律的なスキル選択は現時点では安定せず、実運用で当てにできる水準には至っていない。したがって本設計では、各エージェントが使用する How スキルを CLAUDE.md やサブエージェントプロンプト内で**名指しで明示的に指定する運用を前提**とする。ゆるやかなトリガー条件に頼って自動発火させる設計は採用しない。

ワークフロースキルについては人間が起点となって呼び出すため、この制約の対象外となる。

## 判断基準（How スキルの切り出し）

How スキルについては、以下の基準で切り出しを判断する。ワークフロースキルは人間が呼び出す起点として定型作業ごとに定義されるものであり、本判断の対象には含めない。

- **複数サブエージェント／複数ワークフローから呼び出される汎用的な How** → スキルとして切り出す（DRY 原則）。同じ手法を各エージェントプロンプトやワークフロースキルに重複して書くのではなく、スキルとして一箇所に定義し、各所から参照・使用する。
- **単一サブエージェント固有の How（他のサブエージェントから共有されない）** → 無理にスキルとして切り出す必要はない。そのサブエージェントのプロンプト内に直接含めてよい。過度な切り出しはかえって見通しを悪くし、利用されない孤立スキルを生む。
- **別のプロジェクトに持っていきたい How** → 現時点で単一サブエージェントからの利用であっても、再利用を見越してスキルとして切り出す選択肢がある。

## スキル設計時の注意

- スキルは自己完結させる。プロジェクト固有のドキュメントに依存させない（スキルはドキュメントを参照しない）。
- ワークフロースキルは手順と呼び出し構成の記述が中心。How の詳細は呼び出す How スキル側に委ねる。
