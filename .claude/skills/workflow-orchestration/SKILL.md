# workflow-orchestration — メインエージェントのオーケストレーション契約

## 適用場面

複数のサブエージェントに委譲しながらワークフローを駆動するメイン側に必要な作法を、一括で提供する How スキル。bootstrap-docs / create-workflow / bootstrap-project / improve-workflow が本契約を採用している。

本スキルは **作法**（メインがどう振る舞うか）のみを規定し、**構成**（フェーズが何個並ぶか／何回ループするか／どのフェーズでどのサブを呼ぶか／並列か直列か／観点リストの中身）は一切規定しない。構成は各ワークフロー SKILL.md の責任範囲とする。

設計サイクル（提案レポート → ユーザー承認 → 設計書本体生成）特有の作法は本スキルでは扱わず、`/proposal-design-cycle` に委譲する。本スキルが扱うのは「設計サイクルか否かに関わらずメインが踏むべき横断的な作法」のみ。

## メインエージェントの責務分担

| 集約する責務（メインが担う） | 委譲する責務（サブに任せる） |
|---|---|
| サブエージェント呼び出し（並列／直列の判断） | 設計書本文・成果物本文・レビューレポートの Write |
| 絶対パスの引き回し（`<…>` 中間値の保持） | 本文の生成・編集（Write / Edit の実行） |
| 全 `AskUserQuestion` の発火と回答収集 | レビュー判定・観点別レポート作成 |
| 限定 Read（固定見出しの抜粋）によるユーザー概要提示 | 設計書本文・成果物全文の Read |
| 各ステップの遷移制御・反復制御 | 自処理に閉じた作業の遂行 |

設計サイクル（draft → ユーザー承認 → apply → revise）に伴う responsibilities（設計係の 3 モード契約・承認ゲート手順）は本スキルでは扱わず、`/proposal-design-cycle` に委譲する。各ワークフローは設計サイクルが含まれる場合、`/proposal-design-cycle` も併せて参照する。

## 限定 Read 契約

メインがサブの生成物から Read してよいのは、各ワークフローが定義する「固定見出しテーブル」に列挙された見出しのみ。**本文全読は禁止**。

各ワークフローが SKILL.md 内に下表の 3 カラムテーブルを掲げる。本スキルが規定するのはカラム規約と「本文全読禁止」の規範のみで、見出しの実値は各ワークフローの責任範囲。

| カラム | 用途 |
|---|---|
| 用途 | どのフェーズ／どのユーザー応対のための Read か |
| 対象ファイル | サブが Write した中間生成物の絶対パス（変数表記可） |
| 抜粋する見出し | Read してよい見出し名（Markdown 見出し記法） |

固定見出し以外の本文を Read したくなった場合、それはサブが「メインに渡すべき情報を固定見出しに整形していない」ことを意味する。サブ側の責務として整形を依頼し、メインは限定 Read を破らない。

## 絶対パスの受け渡し記法

各ステップ間で受け渡す中間値は `<…>` 記法で表記する。これは Skill arguments の `$name` 置換やシェル変数ではなく、前ステップのサブエージェント戻り値としてメインが内部で保持し、後続ステップへ Agent 引数として渡す中間値を指す。

- 単一の絶対パス: `<workdir>` `<design-doc>` `<proposal>` 等の名前で内部保持。
- 複数の絶対パス集合: 1 行 1 件の改行区切り文字列で渡す。空集合は空文字列。受け取り側のサブはこの形式を前提に分割・Read・反復処理する。
- ターゲットプロジェクトルート（`<target-project-root>`）は **本スキル発火時の `pwd`** を 1 度スナップショットして以後固定値として扱う。ループ中の `pwd` 変化に追随しない。

## `AskUserQuestion` の取り扱い

- **質問の組み立て・発火作法**（背景の添付／選択肢のメリデメ description／論点 1 件 = 質問 1 件 等）は `/user-hearing` に従う。
- メインは作業開始前に **1 回だけ** `ToolSearch` を `query: "select:AskUserQuestion"`、`max_results: 1` で呼び出してスキーマをロードする。以降のヒアリングはこの 1 回のロード結果を再利用する。
- サブエージェントは `AskUserQuestion` を呼ばない。サブの出力に「このあとユーザーに問うべき論点」が含まれる場合、サブは proposal の固定見出し（`## 提案サマリ` 等）に **推奨値とその根拠を列挙するだけ** で、自らユーザーに問わない。メインがその列挙を限定 Read して `AskUserQuestion` を発火する（設計サイクルでの具体的な発火手順は `/proposal-design-cycle` の approval ゲート契約を参照）。

## サブエージェント戻り値の形式

メインが受け取る戻り値は次の 2 種類のみ。

- **通常のサブエージェント**: 絶対パス 1 行のみ。
- **観点別レビュアー**（`/multi-aspect-review` 契約に従う場合）: `<レポート絶対パス> SP (PASS|FAIL)`（`SP` は半角スペース 1 文字、`PASS|FAIL` はいずれか一方の文字列）。

## 整合チェック失敗時の中断ポリシー

サブエージェントが入力レポート間の値の不一致や、引数の XOR 違反等を検出した場合、以下を遵守する。

- 編集系ツール（`Edit` / 既存対象への `Write` 上書き）を一切呼ばない。
- 不一致内容を自身の出力レポートに列挙して即座に終了する。
- 自動補正・自動進行しない。

中断時の再開はメイン側「失敗時のリカバリ」（各ワークフロー SKILL.md に定義）に従う。サブエージェント側は復旧フローを指示しない。

## サブエージェント実行モード

メインに `AskUserQuestion` を集約しているため、サブエージェント側は **背景実行可否に依存しない**。各ワークフローは全サブを `run_in_background: true` で起動してもよい（戻り値が規約どおり得られればよい）。並列 spawn は単一メッセージ内に複数の `Agent` 呼び出しを並べる方式で実現する。

## メインがしないこと（What NOT to do）

- 設計書本文・成果物本文・レビューレポートの Write
- サブが扱うべき `Write` / `Edit` の直接実行
- 限定 Read 契約で許可された固定見出し以外の本文 Read
- サブエージェントの内部処理手順への介入（メインが渡すのは引数のみ）
- 承認サイクル中の自動進行（ユーザー承認なしに次フェーズへ進まない）

## 既存 How スキルとの組み合わせ方

メイン側の作法は本スキルだけで完結せず、以下の境界で他スキルに委譲する。

| 領域 | 担当スキル | 本スキルが扱うか |
|---|---|---|
| `AskUserQuestion` 1 通の質問本文・選択肢の組み立て作法 | `/user-hearing` | 扱わない |
| 設計サイクル（draft / apply / revise）の 3 モード契約と proposal の構造 | `/proposal-design-cycle` | 扱わない |
| 設計サイクルの approval ゲート手順（1 通の AskUserQuestion で承認 or 修正要望取得） | `/proposal-design-cycle` | 扱わない |
| 観点別レビュアー／再生成 author の契約・PASS/FAIL ループ手順 | `/multi-aspect-review` | 扱わない |
| プロンプト本文の文体（簡潔・明快・一意） | `/prompt-engineering` | 扱わない |
| 横断的なメインの作法（限定 Read 契約・絶対パス引き回し・サブ実行モード・戻り値形式・整合チェック失敗時の中断ポリシー・メインがしないこと） | `workflow-orchestration`（本スキル） | 扱う |

各スキル間で重複が生じないように、責務範囲はこの境界表で明示的に切り分けてある。本スキル内の説明と他スキルの説明が食い違う場合は、各スキル本体を真実の源泉として優先する。

## ワークフロー設計者が決めること（本スキルでは決めないこと）

本契約を採用するワークフローは、以下を自分で定義する。

- (a) ワークフローのステップ構成（何を何個並べるか・何回反復するか・並列か直列か）
- (b) どのサブエージェントを何種類使うか
- (c) 各サブエージェントの内部仕様（プロンプト本体）
- (d) サブ起動の並列／直列の判断
- (e) 限定 Read 契約の固定見出しテーブルの実値（用途／対象ファイル／抜粋見出し）
- (f) サブ起動時の引数仕様（命名・必須/任意の判断）
- (g) `/proposal-design-cycle` を採用する場合、approval ゲートを 1 段にするか 2 段にするか・限定 Read 対象の見出し
- (h) `/multi-aspect-review` を採用する場合、観点リスト `<review-aspects>` の中身
- (i) 中間生成物の命名規約（本スキルは変数表記のみで実値を持たない）
- (j) 失敗時のリカバリ手順
