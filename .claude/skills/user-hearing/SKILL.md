# user-hearing — ユーザーへの問いかけ作法

`AskUserQuestion` でユーザーに判断を仰ぐとき、最小の質問数で・即答可能な質問文に整えるためのリファレンス。エージェントの種類（メイン／サブ）は問わず、`AskUserQuestion` を発火する立場が参照する。

質問文そのものの文体（簡潔・明快・一意）は `/prompt-engineering` の3原則に従う。本スキルは AskUserQuestion 固有の構造（論点・選択肢・自由記述スロット）の組み立て作法に絞る。

## 設計原則

### 1. 判断の必要な論点だけを問う

自動推定や既存資料（調査レポート・ドキュメント等）から確定できる論点は質問にしない。推定値が立つ論点は「推定値で承認 / 別の値を指定」の二択に畳み、ユーザーの確認コストを下げる。

### 2. 質問に背景を添える

「なぜ問うか／答えが何に効くか」を 1〜2 文で示す。ユーザーが追加調査せずに即答できる状態にする。質問文がドキュメント参照だけで完結する形（「proposal を読んでから回答してください」等）にしない。

### 3. 選択肢にメリット・デメリットを併記する

`AskUserQuestion` の `options[].description` に「選んだとき何が起き、何を諦めるか」を書く。`label` だけで判断させない。トレードオフを選択肢自身に内包させることで、ユーザーは選択肢間の比較材料を質問の中で得る。

### 4. 論点 1 件 = 質問 1 件

`AskUserQuestion` 1 通あたり 1 論点を原則とする。1 通に最大 4 件の質問を束ねる仕様だが、束ねるのは **回答が他質問と独立な場合** に限る。後段の答えが前段に依存する場合は順次発火する（前段の選択肢に応じて後段の選択肢が変わるため）。

## AskUserQuestion 発火テンプレート

```
question: <論点を1文で>。<背景: なぜ問うか／答えが何に効くか>
options:
  - label: <選択肢ラベル>
    description: <選んだとき何が起きるか（メリット／デメリット）>
```

実例（中間生成物の永続化形式を選ぶ場面）:

```
question: ヒアリング結果の永続化形式を YAML / JSON のどちらにするか。後段の apply ラウンドで decisions ファイルを Read するため、形式変更は両側に波及する。
options:
  - label: YAML
    description: 人間可読で手修正容易。型表現は弱く、構造が複雑になるとパースが脆くなる。
  - label: JSON
    description: パース堅牢で CI 連携容易。コメント不可で運用メモを残せない。
```

## 自由記述スロットの運用

「別の値を指定」「修正したい」など選択肢では決め切れない論点は、続けて自由記述専用の `AskUserQuestion` を 1 通発火し、具体値を取得する。

- 質問文に「受け付ける入力の種類」を明示する（単一トークンか／複数論点を含む要望か／識別子か等）。
- 選択肢付き `AskUserQuestion` と自由記述専用 `AskUserQuestion` を 1 通に同居させない。回答経路が不明瞭になる。

## アンチパターン

以下に該当したら質問を組み直す。発火前のセルフチェックとして使う。

- 背景なしの裸質問（「Foo を A/B どちらにしますか？」）
- 全選択肢の `description` が同一、または `label` の言い換えになっている（メリデメ差分がない）
- 推定可能な論点を質問にする（例: 調査レポートで確定済みの値を改めて問う）
- 1 通に依存関係のある論点を詰め込む（後段の答えが前段に依存）
- 質問文がドキュメント参照だけで完結する（「proposal.md を確認してから回答してください」）
- 質問数が 7 件を超える（多くなる兆候。判断の必要な論点だけに絞り直す）

## このスキル自身のポリシー

- **本文への混入禁止**: 本スキルの原則・観点は質問を **書く側** が従うものであり、生成する `AskUserQuestion` の `question` / `description` 本文に「メリデメを併記してください」「背景を添えてください」などのメタ的指示を書き込まない。
- **質問文の文体は `/prompt-engineering` に従う**: 簡潔・明快・一意の3原則と「強調語は絶対のルールにだけ使う」を質問本文・description にも適用する。
