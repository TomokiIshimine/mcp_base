# multi-aspect-review — 多角的観点レビューと再生成ループの契約

## 適用場面

複数観点（例: 設計整合性／DRY／規約／機械可読性 など）から成果物群（複数ファイル）を並列点検し、観点別レポートの PASS/FAIL に基づいて該当ファイルだけを並列再生成して合格まで繰り返したいワークフローで参照する。bootstrap-docs / create-workflow がこの契約を採用している。

## 構成する 3 役

品質保証ループは次の 3 種類の役者で構成する。各ワークフローは差分点（観点リスト・author 種別・担当ファイル絶対パス・反復上限）だけを埋めれば、本契約のとおりに動く。

- **観点別レビュアー**（aspect-reviewer）: 1 インスタンスが 1 観点だけを担当し、観点別レポートと PASS/FAIL を返す。
- **再生成 author**（regenerable author）: 1 インスタンスが 1 ファイルだけを担当し、レビューレポートに従って自分の担当ファイルを再生成する。
- **オーケストレーター**（メイン）: 並列 spawn ／ 集約判定 ／ 再生成 dispatch ／ 反復制御を担う。

## 観点別レビュアーの契約

### 入力

- `review-aspect`: 担当する観点識別子（1 つ）
- 対象ファイル絶対パス群（呼び出し元の引数スキーマで定義）
- `output-path`: レビューレポートの書き出し先絶対パス
- `previous-review-path`（任意）: 同観点の前周回レポートの絶対パス（与えられた場合は差分レビューモード）

### 出力

- `<output-path>` にレビューレポートを Write
- 戻り値: `<output-path> PASS` または `<output-path> FAIL`

### 振る舞い

- **担当観点に閉じる**: `review-aspect` で指定された観点の指摘のみ書く。他観点に該当する事項を見つけても本レポートには書かない（他観点を担当する別インスタンスが扱う）。
- **対象ファイルは絶対パスで書く**: 呼び出し元が機械抽出して再生成 dispatch するため。
- **重大度の区分**:
  - 重大 — 担当観点の責務が成立していないレベル
  - 中 — 観点としては成立しているが部分的に逸脱
  - 軽微 — 表現や見出しの揺れなど、自動判定の安定収束を阻害しないレベル
- **PASS/FAIL ルール**: 重大／中が 1 件でもあれば FAIL。軽微のみであれば PASS（軽微はコメントとして残し、反復を不必要に止めない）。
- **差分レビューモード**（`previous-review-path` あり）:
  - 前周回レポートの各指摘について、対象ファイルの該当箇所を Read して解消状況を判定
  - 解消されていない指摘は「未解消」ラベル付きで再掲（指摘文をそのまま引き写す。呼び出し元の機械抽出が壊れないため）
  - 解消された指摘は冒頭の `## 解消済みの前回指摘` に列挙
  - 新規指摘は重大・中を記載する。軽微の新規指摘のみ本周回では記載しない（揺り戻し防止は表現揺れ等の軽微に限定する）。再生成で新たに混入した中・重大の回帰は PASS/FAIL ルールどおり FAIL として扱い、未解消のままループが収束しないようにする（このプロジェクトでの vendor コピー固有の修正。upstream プラグインの「新規は重大のみ記載」から意図的に分岐している）

### レポートテンプレ

各指摘は以下の 5 項目（重大度／対象ファイル／箇所／問題／期待される状態）を必ず揃える。

差分レビュー時のみ `## 解消済みの前回指摘` セクションを含める（新規レビューでは省略）。

```
# レビューレポート（観点: <review-aspect>）

対象: <対象ファイル絶対パス群>
レビュー観点: <review-aspect>
レビュー日時: <ISO8601>

## 解消済みの前回指摘

## 未解消の指摘 / 新規指摘
  - 重大度: <重大|中|軽微>
    対象ファイル: <絶対パス>
    箇所: <見出し名／行番号／frontmatter キー>
    問題: <何が違反・不整合か>
    期待される状態: <どう直すか>

## 判定: <PASS|FAIL>
```

## 再生成 author の契約

### 追加入力

通常の生成入力に加え、再生成時のみ次を受け取る。

- `review-report-path`: レビューレポートの絶対パス（**単数**。配列で渡されない）
- `regeneration-log-path`: 再生成ログの追記先絶対パス

### 振る舞い

`review-report-path` が与えられた場合、author は以下の 4 ステップを実行する。

1. 既存の自分の担当ファイルを Read して現状把握
2. `<review-report-path>` を Read し、`## 未解消の指摘 / 新規指摘` 配下から「対象ファイル絶対パス」が **自分の担当ファイル絶対パス** と一致する指摘のみ抽出。一致しないものは読まない／無視する（疎結合維持）。
3. 抽出した指摘の「期待される状態」を満たすように担当ファイルを Write 上書き。指摘されていない箇所は壊さない。
4. `<regeneration-log-path>` の末尾に追記（既存ログの上書き禁止）:
   - `再生成対象レポート`: レポートファイル名
   - `対応した指摘件数`: N 件
   - `未対応として残した指摘`: あれば、各指摘とその理由

### なぜ単数のレポートなのか

各 author に複数の観点別レポートをマージして渡す設計も可能だが、以下の理由で単数とする。

- author は自分の担当ファイル絶対パスでフィルタするため、複数レポートを渡しても抽出結果は同じ
- 「どのレポートを渡すか」の選択をオーケストレーター契約に閉じ込めることで、author のインターフェースを決定論的に保てる（後述）

## オーケストレーターの契約

### 必要な保持情報

メインは以下を保持していること。

- `<review-aspects>`: 観点識別子のリスト。**順序が API**（後述の `review-report-path` 選択で参照する。並び替えない）。
- `<file-to-author-map>`: 「担当ファイル絶対パス → 起動する author 種別とその引数」の対応関係
- `<iteration-limit>`: 反復上限（推奨: 5 周）
- `<output-path-pattern>`: レビューレポートの命名パターン（例: `<workdir>/06_review-<iteration>_<aspect>.md`）

### ループ手順

`<iteration>` を 1 から始め、(a)〜(d) を 1 周として PASS まで繰り返す。

#### (a) 並列 spawn

`<review-aspects>` の各観点について、観点別レビュアーを **単一メッセージで並列 spawn** する。1 インスタンスに複数観点を任せない。各インスタンスへの引数:

- `review-aspect`: 担当観点識別子（1 つ）
- 対象ファイル絶対パス群
- `output-path`: `<output-path-pattern>` を `<iteration>` と `<aspect>` で具体化したパス
- `previous-review-path`（2 周目以降のみ）: 同観点の前周回レポート絶対パス。観点をまたいで渡さない。

#### (b) 集約判定

各レビュアーの戻り値（`<output-path> PASS` / `<output-path> FAIL`）を集約する。

- 全観点が `PASS` ならループを抜ける（合格 → 後続処理へ）
- 1 観点でも `FAIL` があり `<iteration> < <iteration-limit>` なら (c) へ
- 1 観点でも `FAIL` があり `<iteration> == <iteration-limit>` なら反復上限到達 → ループ脱出（詳細は後述「反復上限到達時の扱い」）

#### (c) 再生成フロー

1. FAIL 観点のレポートから `## 未解消の指摘 / 新規指摘` 配下の「対象ファイル絶対パス」を全て抽出し、重複排除する。
2. 各 author に渡す `review-report-path` を **1 本だけ** 選ぶ:
   - 候補は **その author の担当ファイルを FAIL 指摘した観点の今 iteration レポート群**
   - 複数候補がある場合は `<review-aspects>` の **宣言順で最初に現れた観点** を選ぶ（決定論的）
   - 選んだ 1 本のレポートのみを author に渡す（配列で渡さない）
3. `<file-to-author-map>` を引いて、再生成すべき author を起動する（複数 author を起動する場合は単一メッセージで並列 spawn）。

#### (d) 反復制御

`<iteration>` を 1 増やして (a) へ戻る。

### 反復上限到達時の扱い

反復上限到達は失敗ではなく、その時点の状態で打ち切ってユーザーに報告する。ユーザーに「最大反復回数に達しました」と警告し、最終周回のレポート絶対パス一覧を渡して手動修正できるようにする。自動でリトライしない。

## ワークフロー設計者が決めること

本契約を採用するワークフローは、以下を自分で定義する。

| 決定項目 | 例（bootstrap-docs） | 例（create-workflow） |
|---|---|---|
| 観点リスト `<review-aspects>` | `design-fidelity / dry / min-context / decoupling` | `design-fidelity / context-engineering / convention / machine-readable / prompt-engineering` |
| 各観点の責務定義 | `bootstrapdocs-reviewer` 内 | `workflow-reviewer` 内 |
| 再生成 author 種別と担当ファイル | docs/readme/claudemd の 3 体 | agent/skill の 2 体 |
| 反復上限 `<iteration-limit>` | 5 | 5 |
| レビューレポートの命名規約 | `<workdir>/06_review-<iteration>_<aspect>.md` | 同左 |

## このスキル自身のポリシー

- **サブエージェント定義への転写禁止**: 各役者の振る舞いを定義に転写せず、定義側からは「`/multi-aspect-review` の <役者> 契約に従う」と参照する形で再利用する。
- **契約変更時の波及確認**: 本スキルの契約を変更する場合は、参照している全ワークフロー（bootstrap-docs / create-workflow / 他）の整合性を確認する。
