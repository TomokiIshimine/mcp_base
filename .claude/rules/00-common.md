# 共通コーディング規約（全 Python コード）

src/・tests/ を問わず、すべての Python コードに適用する普遍的な規約。
リンター/フォーマッタ（ruff format・ruff lint・mypy strict・bandit）が機械的に
強制する範囲（整形・import 整列・未使用検出・近代化・型注釈の付与と整合・命名の
ケース）はここには書かない。本ファイルはツールが判定できない設計判断のみを扱う。

本ファイルは `paths` を持たず、起動時に常時読み込む（新規ファイルを白紙から
書く場面でも普遍規約が必ず文脈に入るようにするため）。層別の具体規約（05〜60）
は該当ファイルを読んだ時に発火する path-scoped ルールとして分離する。

## 新規ファイル作成時

- `src/`・`tests/` 配下に**新規ファイルを作る前に**、その配置パスに対応する
  `.claude/rules/`（層別: 05〜60）を読んでから書く。白紙からの `Write` では層別
  規約が文脈に入らないため、作成前に自分で参照して規約を満たす。

## 用語と命名

- ユビキタス言語を全層で統一する。同一概念には同一語を用いる（例: `greeting`）。
  層をまたいで別名・同義語に言い換えない。
- 名前は意図を表す。型・件数・単位・役割が読み取れる名前にし、`data` `info`
  `tmp` `obj` のような無情報な名前を避ける。
- 真偽値は肯定形の述語にする（`is_valid` / `has_id`）。否定の二重化を避ける。
- 略語は広く通用するもの（id, db, sql, url）に限る。独自の短縮を作らない。
- 命名のケース（snake_case / PascalCase / 定数 UPPER_CASE）は ruff の N ルールが
  強制する。本規約では再掲しない。

## ドキュメンテーション・コメント

- 公開する関数・クラス・モジュールの非自明な意図には docstring を書く。
  シグネチャを見れば分かることの言い換えは書かない。
- コメントは「何を」ではなく「なぜ」を説明する。制約・前提・トレードオフ・将来の
  注意点を残す。コードと矛盾するコメントを放置しない。
- コメント・docstring は日本語で書いてよい（本プロジェクトの方針）。

## モジュールの責務

- 1 モジュール = 1 責務。無関係な関心を同居させない。
- 公開 API は意図して絞る。モジュール外から使わせない実装詳細は先頭 `_` で明示する。
