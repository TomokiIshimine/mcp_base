# アーキテクチャ横断規約（本番コード）

src/ 配下の本番コード全層に共通する、クリーンアーキテクチャを支える設計判断。
各層に固有の具体化は各層のルール（10〜50）で扱い、ここでは横断原則のみを定める。

## 依存方向

- 依存は内側へのみ向かう。外側は内側を知り、内側は外側を知らない。
  - `interface_adapter` → `usecase` → `domain`
  - `infrastructure` → `usecase`（ポート）・`domain`
- 内側の層（domain・usecase）から外側の層（infrastructure・interface_adapter・
  framework）を import しない。
- 層をまたぐ具象の生成・結線は合成ルート（`app.py`）に集約する。各層は自分で
  下位の具象を組み立てない。

## エラーハンドリング

- 例外は層境界で変換する。下位層の例外型をそのまま上位へ素通しさせず、その層の
  契約上の例外型に翻訳して送出する（層別の例外型は各層ルール参照）。
- 例外を握り潰さない。捕捉したら、変換して再送出するか、記録して回復する。
  握り潰しは回帰テストで検知できるよう意図を明示する。
- 設定不備・前提崩れは fail-fast。不完全な状態で処理を続行しない。

## 型設計

- `Any` を使わない。mypy strict が許す逃げ道（明示的 `Any`・`cast` での回避）に
  頼らず、正確な型で表現する。
- プリミティブ偏重を避ける。意味を持つ値は domain の値オブジェクト/エンティティで
  表し、`str`/`int` の裸の受け渡しでドメイン概念を表現しない。
- 既定は不変。可変状態は必要な箇所に限定する。

## ロギング

- ロガーは `infrastructure.logging_config` 経由で取得する。`print` で記録しない。
- 同一事象を複数層で二重に記録しない。記録責務を 1 層に定める（層別責務は各層
  ルール参照）。成功は usecase が、失敗は呼び出し側の上位層が記録する、という
  役割分担を崩さない。
- 機微情報（資格情報・接続文字列・個人情報）をログに出さない。
- ログレベルは意味で選ぶ。正常系の業務イベント=INFO、回復可能な異常=WARNING、
  処理失敗=ERROR、調査用の技術詳細=DEBUG。
