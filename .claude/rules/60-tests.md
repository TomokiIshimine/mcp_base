# テストの規約

pytest 設定（testpaths・マーカー登録・カバレッジ）やリンター/フォーマッタが
カバーする範囲は除外し、判断が要る方針のみを定める。

## 配置・命名

- `tests/` は `src/` の層構成をミラーする（domain / usecase / infrastructure /
  interface_adapter / integration）。
- テストファイル名は対応する本番モジュールが分かる形にする（`test_<module>.py`）。
  ケースは ruff の N ルールが強制するため再掲しない。

## 層別のテストスタイル

- domain: 純データの検証。フェイク不要。
- usecase: ポートのフェイクを注入して検証し、実 DB に接続しない。
- infrastructure: 軽量テスト（フェイクプール/モックで例外翻訳・SQL を検証）と、実
  MySQL を要する統合テストを分離する。
- interface_adapter: usecase をモックし、描画分岐・失敗時の挙動を検証する。

## 軽量 / 統合の分離

- 実 MySQL を必要とするテストは `@pytest.mark.integration` を付け、既定の軽量実行
  から除外する。軽量テストは外部依存（DB・ネットワーク）を立てない。

## テストの質

- 1 テスト = 1 関心。Arrange-Act-Assert の構造を保つ。
- 実装の内部詳細ではなく、観測可能な振る舞い・契約を検証する。
- テストは順序・共有状態に依存しない。各テストが前提を自前で用意する。

## フェイク・フィクスチャ

- 複数テストで共有するフェイク/フィクスチャは置き場と命名を統一し、本番の型・
  ポート契約に追従させる。
