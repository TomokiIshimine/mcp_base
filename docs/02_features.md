# 機能仕様

本プロジェクトが提供する機能の一覧と、各機能の入出力・前提・正常系の振る舞いを定義する。対象は挨拶レコード（greeting）の CRUD と、その全体を保護する Google OAuth2 認証ゲート。本ドキュメントは「何が正常完了か」を定義し、画面の操作詳細は `docs/03_screens.md`、データの属性・制約は `docs/04_data-design.md`、環境変数・運用手順（管理者 Email・OAuth クライアント設定の具体）は `docs/06_operations.md`、受入シナリオは `docs/05_e2e-tests.md` を参照する。

プロジェクトの位置づけ（名称 `mcp-base` と現行実態の関係・将来構想）は `docs/01_architecture.md` の「プロジェクトの位置づけ」を参照。

## 機能一覧

| 機能 | 概要 | 対応画面（`docs/03_screens.md`） |
|---|---|---|
| 認証・認可 | Google OAuth2 でログインし、管理者 Email と一致する利用者のみに CRUD 全体の利用を許可する | ログイン導線・拒否画面 |
| 一覧表示 | 登録済みの挨拶レコードを id 昇順で全件表示する | 一覧 |
| 作成 | メッセージを入力して挨拶レコードを 1 件追加する | 新規作成 |
| 更新 | 既存レコードのメッセージを書き換える | 更新 |
| 削除 | 既存レコードを 1 件削除する | 削除 |

挨拶レコードは id（採番された整数）と message（文字列）からなる。属性・型・永続化スキーマは `docs/04_data-design.md` を参照。

## メッセージの共通制約

作成・更新で受け付けるメッセージは次を満たす。満たさない入力は拒否し、レコードは変更されない。

- 前後の空白を除去した結果が空でないこと。
- 前後の空白を除去した結果が 255 文字（Unicode コードポイント数）以内であること。

メッセージは保存前に前後の空白が除去される。文字数上限の根拠と算出方法の詳細は `docs/04_data-design.md` を参照。

## 認証・認可

アプリ全体を Google を ID プロバイダとする OAuth2 認証ゲートで保護する。認証・認可を満たさない限り、CRUD 機能（一覧表示・作成・更新・削除）は一切描画しない。ユビキタス言語として、認証で得る Google アカウントのメールアドレスを `email`、環境変数で登録する許可対象を「管理者 Email」と呼ぶ。

- 入力: 利用者の Google アカウントによるログイン（認証の結果として `email` と `email_verified` を取得する）。許可対象の管理者 Email は環境変数で 1 件登録する（変数名・設定手順は `docs/06_operations.md` を参照）。
- 出力: 認可結果に応じた 3 分岐の画面（未認証時のログイン導線／認可不成立時の拒否画面／認可成立時の CRUD 画面）。各画面の操作詳細は `docs/03_screens.md` を参照。
- 前提: 管理者 Email の環境変数が設定済みであること。未設定・空文字の場合はアプリ起動時に停止し（fail-fast）、CRUD 画面を描画しない。
- 正常系の振る舞い:
  - 未認証の利用者にはログイン導線を提示し、CRUD は描画しない。
  - 認証で得た `email_verified` が true、かつ `email` が管理者 Email とバイト単位で完全一致する場合のみ認可成立とし、CRUD を描画する。大文字小文字の無視・前後空白の除去などの正規化は行わない。
  - 認証成功でも認可不成立（管理者 Email と不一致、または `email_verified` が true でない）の場合は、拒否メッセージと別アカウントでの再ログイン導線を表示し、CRUD は描画しない。
  - 認可済み画面・拒否画面の双方にログアウト導線を置く。ログアウト後は未認証状態へ戻る。
  - 認証成功・認証拒否（未認証・Email 不一致・未検証）をログに記録する。`email` をはじめとする機微情報は平文で出力せずマスクする。
- 関連: 認可判定ユースケース `AuthorizeAdminUseCase`（`src/usecase/authorize_admin_usecase.py`）／認証ゲート view `render_auth_gate`（`src/interface_adapter/auth_gate_view.py`）／受入シナリオ `docs/05_e2e-tests.md`。

## 一覧表示

- 入力: なし。
- 出力: 全挨拶レコード（id・message）を id 昇順で並べた一覧。
- 前提: なし。
- 正常系の振る舞い:
  - レコードが 1 件以上あれば、全件を id・message の表として表示する。
  - レコードが 0 件であれば、レコードがない旨を表示し、作成フォームへ誘導する。
- 関連: ユースケース `list_greetings`（`src/usecase/manage_greetings_usecase.py`）／受入シナリオ `docs/05_e2e-tests.md`。

## 作成

- 入力: メッセージ（文字列）。
- 出力: 採番済みの挨拶レコード（id・message）。作成後、一覧に当該レコードが現れる。
- 前提: メッセージが「メッセージの共通制約」を満たす。
- 正常系の振る舞い: メッセージを前後空白除去のうえ新規レコードとして保存し、id を採番する。保存後は一覧へ反映される。
- 関連: ユースケース `create_greeting`（`src/usecase/manage_greetings_usecase.py`）／受入シナリオ `docs/05_e2e-tests.md`。

## 更新

- 入力: 対象レコードの id、新しいメッセージ（文字列）。
- 出力: なし（対象レコードの message が新しい値へ置き換わる）。
- 前提:
  - 対象 id のレコードが存在する。
  - 新しいメッセージが「メッセージの共通制約」を満たす。
- 正常系の振る舞い: 指定 id のレコードの message を、前後空白除去後の新しい値へ更新する。id は変更しない。更新後は一覧へ反映される。
- 関連: ユースケース `update_greeting`（`src/usecase/manage_greetings_usecase.py`）／受入シナリオ `docs/05_e2e-tests.md`。

## 削除

- 入力: 対象レコードの id。
- 出力: なし（対象レコードが一覧から消える）。
- 前提: 対象 id のレコードが存在する。
- 正常系の振る舞い: 指定 id のレコードを削除する。削除後は一覧へ反映される。
- 関連: ユースケース `delete_greeting`（`src/usecase/manage_greetings_usecase.py`）／受入シナリオ `docs/05_e2e-tests.md`。
