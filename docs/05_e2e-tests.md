# E2E 受入シナリオ（正常系）

挨拶（greeting）レコードの CRUD が利用者の操作で end-to-end に成立することを確認する受入シナリオの一覧。各シナリオは「前提・操作・期待結果（受入条件）」で構成し、すべて満たせば受入 OK とする。

- 範囲: 正常系のみ。異常系（入力不正・対象不在・DB 障害・接続枯渇など）は本ドキュメントの対象外。
- 対象経路: Streamlit 画面 → controller → usecase → MySQL リポジトリ → `greetings` テーブルまでを通した実経路。
- 各機能の入出力・前提・正常系の振る舞いの定義は `@docs/02_features.md` を正とし、本ドキュメントは受入の観点（何が通れば OK か）に徹する。
- テストの書き方（AAA・1 テスト 1 関心・軽量／統合テストの分離・フェイクの置き場など）は `.claude/rules/60-tests.md` を参照する。本ドキュメントには記載しない。

## 前提環境（全シナリオ共通）

- 起動済みの MySQL と `greetings` テーブルが存在する（テーブル定義は `docs/04_data-design.md`、起動手順は `docs/06_operations.md`）。
- Streamlit アプリが起動済みで、挨拶 CRUD 画面（タイトル `Greetings CRUD`）が表示できる（画面の構成は `docs/03_screens.md`）。
- 本節の CRUD シナリオ（E2E-01〜06）は認証ゲートを通過した認可済みセッションを前提とする。CRUD 画面は管理者 Google アカウントでの認可成立後にのみ描画されるため、各シナリオは「認証 E2E 受入シナリオ」節の AUTH-03 が成立した状態を開始点とする。認証ゲート自体の受入は同節で扱う。
- 各シナリオは開始時の `greetings` テーブル状態を前提に明記し、他シナリオの実行順序に依存しない。

## シナリオ一覧

| ID | シナリオ | 対応機能（`@docs/02_features.md`） |
|---|---|---|
| E2E-01 | 空状態の初期表示 | 参照／一覧 |
| E2E-02 | 挨拶を作成する | 作成 |
| E2E-03 | 一覧で作成済みの挨拶を確認する | 参照／一覧 |
| E2E-04 | 挨拶を更新する | 更新 |
| E2E-05 | 挨拶を削除する | 削除 |
| E2E-06 | 作成→一覧→更新→削除の一連フロー | 作成・参照／一覧・更新・削除 |

## E2E-01 空状態の初期表示

- 前提: `greetings` テーブルにレコードが 0 件。
- 操作: 挨拶 CRUD 画面を開く。
- 期待結果:
  - 一覧領域に「レコードがありません。下のフォームから作成してください。」の案内が表示される。
  - 新規作成フォーム（メッセージ入力欄と「作成」ボタン）が表示される。
  - 更新・削除の操作 UI は表示されない（対象 id が存在しないため）。

## E2E-02 挨拶を作成する

- 前提: 挨拶 CRUD 画面を開いている。
- 操作: 新規作成フォームのメッセージ欄に `hello` を入力し、「作成」ボタンを押す。
- 期待結果（受入条件）:
  - 一覧に作成した `hello` が 1 件表示される。
  - 作成行に DB 採番の id（正の整数）が付与されている。
  - メッセージ入力欄が空にクリアされている（`clear_on_submit=True` による挙動）。
- 注記: 成功メッセージ「作成しました。」は送信直後に一瞬表示されるが、直後の再描画（`st.rerun()`）で安定画面には残らないため受入条件にしない。

## E2E-03 一覧で作成済みの挨拶を確認する

- 前提: `greetings` テーブルに 2 件以上のレコードが存在する。
- 操作: 挨拶 CRUD 画面を開く。
- 期待結果:
  - 一覧に全レコードが id 昇順で表示される。
  - 各行に id と message の 2 列が表示される。
  - 表示件数が `greetings` テーブルの件数と一致する。

## E2E-04 挨拶を更新する

- 前提: 挨拶が 1 件以上存在し、更新対象の id とその現在のメッセージが分かっている。
- 操作: 更新フォームで対象 id を選択し（メッセージ欄には選択 id の現在値が初期表示される）、新しいメッセージ `updated` に書き換えて「更新」ボタンを押す。
- 期待結果（受入条件）:
  - 一覧の対象 id の message が `updated` に変わる。
  - 対象 id は変わらない。他レコードのメッセージは変わらない。
- 注記: 成功メッセージ「更新しました。」は送信直後に一瞬表示されるが、直後の再描画（`st.rerun()`）で安定画面には残らないため受入条件にしない。

## E2E-05 挨拶を削除する

- 前提: 挨拶が 1 件以上存在し、削除対象の id が分かっている。
- 操作: 削除 UI で対象 id を選択し、「削除」ボタンを押す。
- 期待結果（受入条件）:
  - 一覧から対象 id の行が消える。
  - 残りのレコードは id・message ともに保持される。
  - 削除後に 0 件になった場合は E2E-01 と同じ空状態表示に戻る。
- 注記: 成功メッセージ「削除しました。」は送信直後に一瞬表示されるが、直後の再描画（`st.rerun()`）で安定画面には残らないため受入条件にしない。

## E2E-06 作成→一覧→更新→削除の一連フロー

CRUD の各機能が同一データに対して連続して整合することを確認する通し受入。

- 前提: `greetings` テーブルにレコードが 0 件。
- 操作と期待結果（各ステップの受入条件は再描画後の一覧反映で判定する）:
  1. メッセージ `hello` を作成する → 一覧に `hello` が 1 件（id 付き）表示される。
  2. 一覧を確認する → 1.で作成した id・message が表示される。
  3. その id を `hello updated` に更新する → 一覧で同 id の message が `hello updated` に変わる。
  4. その id を削除する → 一覧が 0 件になり空状態表示へ戻る。
- 受入条件: 上記 1〜4 が中断なく順に成立し、各ステップの結果が一覧表示に反映されている。
- 注記: 各操作の成功メッセージ（「作成しました。」等）は送信直後に一瞬表示されるが、直後の再描画（`st.rerun()`）で安定画面には残らないため受入条件にしない。

# 認証 E2E 受入シナリオ

挨拶 CRUD 画面の手前に Google アカウント認証ゲートが入る。本節は認証ゲートが利用者の操作で end-to-end に成立することを確認する受入シナリオの一覧。各シナリオは「前提・操作・期待結果（受入条件）」で構成し、すべて満たせば受入 OK とする。

- 範囲: 認証ゲートの 3 分岐（未認証 → ログイン導線／認可不成立 → 拒否画面 ＋ 再ログイン導線／認可成立 → CRUD 描画）、起動時の設定 fail-fast、認証イベントログの Email マスクまで。拒否分岐・起動 fail-fast は認証ゲートの定義された受入挙動であり本節の対象に含む（入力不正・DB 障害などの異常系とは区別する）。
- 認可成立条件: ログイン済み（`is_logged_in=True`）かつ `email_verified=True` かつ Google アカウントの `email` が管理者 Email とバイト単位で完全一致。正規化（大文字小文字の無視・前後空白の除去）はしない。
- 管理者 Email の登録経路・OAuth クライアント設定・起動手順は `docs/06_operations.md`、画面（ログイン導線・拒否画面・ログアウト導線）の構成は `docs/03_screens.md` を正とし、本節は受入の観点（何が通れば OK か）に徹する。
- テストの書き方（AAA・1 テスト 1 関心・軽量／統合テストの分離・フェイクの置き場など）は `.claude/rules/60-tests.md` を参照する。本ドキュメントには記載しない。

## 完全 E2E の制約と代替方針（フェイク st 結合テスト）

- 制約: AUTH-01〜AUTH-08 の完全 E2E は実 Google OAuth クライアント資格（`client_id` / `client_secret`）と対話的な Google 同意画面を要する。実行環境に実資格情報が無く、同意フローは外部サイトへの遷移・モーダルを伴うため、ブラウザ自動操作では完遂できない（環境依存）。
- 代替方針: 実装の設計判断により、上記制約下では `st.user` をフェイク／スタブで差し替えた結合テストで受入挙動を代替検証することを認める。ゲート 3 分岐・描画分岐（AUTH-02〜AUTH-07 相当）は `tests/interface_adapter/test_auth_gate_view.py`、認可判定（完全一致・`email_verified` 境界・正規化なし）は `tests/usecase/test_authorize_admin_usecase.py`、起動 fail-fast（AUTH-01）は `tests/infrastructure/test_auth_config.py` の設定テスト、Email マスク（AUTH-08）は `tests/infrastructure/test_email_mask.py` で担保する。
- 受入判定: 完全 E2E を実機実行できない環境では、上記のフェイク st 結合テスト・認可判定／設定／マスクユニットがすべて緑であることをもって AUTH-01〜AUTH-08 の受入充足とみなす。

## シナリオ一覧

| ID | 対応 AC | シナリオ | 主な検証手段（代替方針） |
|---|---|---|---|
| AUTH-01 | AC-1 | 管理者 Email 未設定（または空文字）で起動 | `tests/infrastructure/test_auth_config.py`（設定 fail-fast） |

| AUTH-03 | AC-3 | 管理者 Email と完全一致・検証済みアカウントでログイン | `test_authorize_admin_usecase.py`（認可成立）＋ `test_auth_gate_view.py`（CRUD 描画分岐） |
| AUTH-04 | AC-4 | 管理者 Email と一致しないアカウントでログイン | `test_authorize_admin_usecase.py`（不一致拒否）＋ `test_auth_gate_view.py`（拒否分岐） |
| AUTH-05 | AC-5 | `email` 一致だが `email_verified` が true でない | `test_authorize_admin_usecase.py`（検証フラグ境界）＋ `test_auth_gate_view.py` |

## AUTH-01 設定不備での起動 fail-fast

- 前提: 管理者 Email の環境変数が未設定、または空文字。
- 操作: アプリを起動する。
- 期待結果（受入条件）:
  - CRUD 画面を描画せず、起動時に停止する（`ConfigError` 系での fail-fast）。
  - 認証ゲート以降（ログイン導線・CRUD）のいずれも表示されない。

## AUTH-02 未認証時のログイン導線

- 前提: 未認証（ログインしていない）状態。
- 操作: アプリ URL を開く。
- 期待結果（受入条件）:
  - CRUD 画面は描画されない。
  - Google ログイン導線（`st.login()` を起動するボタン）が提示される。

## AUTH-03 管理者アカウントでの認可成立

- 前提: 管理者 Email と完全一致し `email_verified=true` の Google アカウントが利用できる。
- 操作: そのアカウントでログインする。
- 期待結果（受入条件）:
  - CRUD 画面（タイトル `Greetings CRUD`）が表示され、CRUD 操作ができる。
  - 認可済み画面にログアウト導線が表示される。
- 注記: 本シナリオの成立後の状態が、前節 CRUD シナリオ（E2E-01〜06）の開始前提（認可済みセッション）に対応する。

## AUTH-04 管理者以外のアカウントでの拒否

- 前提: 管理者 Email と一致しない Google アカウントが利用できる。
- 操作: そのアカウントでログインする。
- 期待結果（受入条件）:
  - 拒否メッセージと、別アカウントでの再ログイン導線（ログアウト起点）が表示される。
  - CRUD は一切描画されない（CRUD 内容が漏れない）。

## AUTH-05 未検証 Email での拒否

- 前提: `email` は管理者 Email と一致するが `email_verified` が true でないアカウントが利用できる。
- 操作: そのアカウントでログインする。
- 期待結果（受入条件）:
  - AUTH-04 と同じ拒否状態になる（拒否メッセージ ＋ 再ログイン導線、CRUD 非描画）。
  - `email_verified` が True 以外（False / 欠落 / None）はすべて拒否として扱う。

## AUTH-06 正規化なしによる不一致拒否

- 前提: 管理者 Email と大文字小文字のみ、または前後空白のみが異なる `email` のアカウントが利用できる。
- 操作: そのアカウントでログインする。
- 期待結果（受入条件）:
  - 不一致として拒否される（正規化しない）。バイト単位の完全一致のときのみ認可成立。

## AUTH-07 ログアウトによる未認証復帰

- 前提: 認可済み画面、または拒否画面のいずれかが表示されている。
- 操作: その画面のログアウト導線（`st.logout()` 起点）からログアウトする。
- 期待結果（受入条件）:
  - ログアウト後は未認証状態（AUTH-02 相当：ログイン導線が提示され CRUD 非描画）へ戻る。

## AUTH-08 認証イベントログの Email マスク

- 前提: 認証成功・認証拒否（未認証／Email 不一致／未検証）を発生させられる。
- 操作: 認証成功と認証拒否をそれぞれ発生させ、ログ出力を確認する。
- 期待結果（受入条件）:
  - 認証成功・認証拒否がログに記録される。
  - ログ中の `email` が平文でなくマスク済み（`a***@example.com` 形式：先頭 1 文字を残しローカル部をマスク、ドメインは保持）で出力される。
  - 管理者 Email・`client_secret`・`cookie_secret` の平文がログ・例外・repr のいずれにも現れない。
