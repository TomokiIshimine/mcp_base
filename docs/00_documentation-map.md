# ドキュメントマップ

`docs/` 配下のドキュメント体系全体の索引と設計意図をまとめたカタログ。各ドキュメントの目的・対象読者・更新契機と、ドキュメント間の責務境界をここで一元管理する。個別の仕様・設計内容は各ドキュメント本体に置き、本ドキュメントには再掲しない。

## docs/* カタログ

| ファイル | 目的 | 対象読者 | 更新契機 |
|---|---|---|---|
| `docs/00_documentation-map.md` | docs 体系全体の索引・責務境界・更新ポリシー（本ドキュメント） | 開発者・Claude Code 全エージェント | docs の追加・削除・責務変更時 |
| `docs/01_architecture.md` | 4 層構成と現行ファイルの責務・依存方向の地図、プロジェクトの位置づけ | 開発者・Claude Code 全エージェント | 層構成・主要ファイル・依存方向の変更時 |
| `docs/02_features.md` | 提供機能（greeting レコードの CRUD、Google OAuth2 ログイン・管理者 Email 認可）の機能仕様 | 開発者・利用者 | 機能の追加・変更・廃止時 |
| `docs/03_screens.md` | Streamlit が描画する画面の一覧・役割・操作・遷移 | 開発者・利用者 | 画面・UI 操作の追加・変更時 |
| `docs/04_data-design.md` | ドメインモデルと MySQL 永続化スキーマの設計 | 開発者 | ドメインモデル・テーブル定義・対応関係の変更時 |
| `docs/05_e2e-tests.md` | E2E 受入シナリオ一覧（CRUD 正常系と、拒否分岐・起動時 fail-fast を含む認証ゲートの受入条件） | 開発者・QA | 受入シナリオ・対応機能の変更時 |
| `docs/06_operations.md` | 起動・MySQL/Docker・環境変数・統合テストの運用手順 | 利用者・開発者 | 起動手順・運用資材・環境変数の変更時 |

## 読み順

- 全体像を把握する: `docs/01_architecture.md`
- 何ができるかを知る: `docs/02_features.md` → `docs/03_screens.md`
- データ構造を知る: `docs/04_data-design.md`
- 動かす: `docs/06_operations.md`
- 受入条件を確認する: `docs/05_e2e-tests.md`

プロジェクトの位置づけ（名称 `mcp-base` と現行実態が Streamlit 製 greeting CRUD である関係、将来の MCP サーバー化構想）の正本は `docs/01_architecture.md` の「プロジェクトの位置づけ」にある。他ドキュメントはそこを参照する。

## 3 層の責務境界

ドキュメントは CLAUDE.md・README.md・docs/* の 3 層に分かれ、各層の責務は次のとおり重複しない。

| 層 | 責務 | 対象読者 | 含めないもの |
|---|---|---|---|
| `CLAUDE.md` | Claude Code 全エージェント共通の基盤（プロジェクト目的・設計原則の要約・技術スタック概要・ディレクトリ構成・索引へのパス） | Claude Code 全エージェント | 詳細設計・運用手順・規約本文（docs/* と `.claude/rules/` へパス参照） |
| `README.md` | 利用者向け入口（プロジェクト概要・セットアップ・起動・主要ドキュメントへのリンク） | 利用者・開発者 | Claude Code 専用規約・詳細運用・詳細設計 |
| `docs/*` | 専門領域ごとの仕様・設計地図・データ・画面・受入条件・運用手順 | 開発者・利用者・QA・Claude Code 全エージェント | 全エージェント共通基盤（CLAUDE.md）・利用者向け入口（README.md）・コーディング規約本文（`.claude/rules/`） |

索引は本ドキュメント（`docs/00_documentation-map.md`）に集約し、CLAUDE.md・README.md からはここへパス参照で誘導する。

## `.claude/rules/` との MECE 方針

コーディング規約は `.claude/rules/`（`.claude/rules/00-common.md` ほか層別 05〜60）に集約済みで、docs/* とは次のように責務を分ける。

- `.claude/rules/` = How（コードの書き方・実装上の制約）。
- `docs/*` = What・地図（仕様・設計・データ・画面・受入条件・運用手順）。

両者にまたがる情報は規約側を正本とし、docs/* からは規約本文を再掲せず相対パスで参照する。例として、依存方向や例外翻訳の制約は `docs/01_architecture.md` から `.claude/rules/05-architecture.md` を、永続化実装の制約は `docs/04_data-design.md` から `.claude/rules/30-infrastructure.md` を、テストの書き方は `docs/05_e2e-tests.md` から `.claude/rules/60-tests.md` を参照する。

## ドキュメント間参照の記法

ドキュメント本文から他ファイルを参照するときは、依存の強さで記法を使い分ける。

- **密依存（必ず参照する関係）**: `@` を付けた相対パス（例 `@docs/02_features.md`）で書く。Claude Code が参照元を読むと `@` 先を自動でコンテキストにロードする。参照元を理解するうえで対象ファイルの内容が常に必要な関係にのみ使う。
- **疎依存（必要に応じて参照する関係）**: プレーンな相対パス（例 `docs/02_features.md`）で書く。読み手が必要と判断したときだけ開く。

`@`（自動ロード）は最小コンテキストとのバランスで密依存にのみ使い、疎依存には付けない。現状 `@` を付けているのは次の 2 箇所のみで、他のドキュメント間参照はすべてプレーンとする。

| 参照元 | `@` 参照先 | 密依存とする理由 |
|---|---|---|
| `CLAUDE.md` | `@docs/00_documentation-map.md` | 全エージェントが索引（本ドキュメント）を常に要する |
| `docs/05_e2e-tests.md` | `@docs/02_features.md` | 受入シナリオは機能仕様に常に紐づく |

本ドキュメントが上記カタログ・読み順で列挙する docs/01〜06 への参照はプレーンのまま（`@` を付けない）とする。カタログを開いただけで全 docs が自動ロードされるのを避け、最小コンテキストを保つため。

## 更新ポリシー

- ドキュメントは現行コードを正とし、コードに追従して更新する。コードと乖離した記述は残さない。
- 各ドキュメントは「地図」に徹する。コード全体を読まずに全体像を把握し、目的のコードへ到達するための索引として保つ。
- 1 つの情報は 1 つのドキュメントにのみ置く。他ドキュメントの内容は再掲せず、相対パスで参照する。
- ドキュメントの追加・削除・責務変更時は、本ドキュメントのカタログ・責務境界・読み順を併せて更新する。
